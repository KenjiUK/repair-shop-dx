**作成日**: 2025-01-XX  
**目的**: 外部AIからの見解をレビューし、提案書を改善

## 📋 外部AIからの見解の要約

### 核心的な洞察

**整備工場の業務は「3つの入口 × 2つの判断 × 2つの分岐」で説明できる**

#### ① お客さんの「入庫前パターン」は最低3つある

**パターンA: 「とにかく車検を通してほしい」型（即入庫）**
- 典型的な会話: 「車検お願いします。あとは見てください」
- 業務フロー: 受付 → 法定車検前提の最低限見積（仮） → 入庫・受入れ点検 → 問題が出たら法定NG/任意 → 電話・LINEで承諾 → 作業・完成
- **この人は「決断を工場に委ねている」**

**パターンB: 「事前に見積が欲しい」型（仮チェックあり）**
- 典型的な会話: 「いくらかかるか見てから決めたい」
- 業務フロー: 受付 → 事前点検（簡易 or 仮） → 見積提示 → 承諾 → 入庫・作業 → 作業中に問題発見 → 追加見積 → 承諾
- **「2回目・3回目の見積」が起こる**

**パターンC（重要）: 「困りごと起点」型（車検はきっかけ）**
- 典型的な会話: 「車検もだけど、最近◯◯が気になる」（エアコン、異音、ハンドルブレ、警告灯）
- **車検は"入口"で、本当の仕事は別**

#### ② 整備工場の中では「2種類の判断」が常に走っている

**判断①: これは「法的に必須」か？**
- 車検不適合、保安基準NG、通らない
- **お客さんの同意は形式的（やらない選択肢がない）**

**判断②: これは「商売として提案」か？**
- 今は通る、でも危ない・近い・次はダメ
- **お客さんの意思決定が必要**

#### ③ さらに「2つの分岐」がある（ここが混乱の元）

**分岐A: 車検という枠の中で完結させるか？**
- ブレーキ、タイヤ、オイル、ワイパー
- **車検と一体で説明できる**

**分岐B: 車検とは切り分けるべきか？**
- エアコン、電装、内装、快適装備
- **説明・予算・判断軸が別**

### 業務の本質

**整備工場の業務は、実はこうです：**

> 「車を直す」ではなく  
> 「お客さんの意思決定を、段階的に支援する」仕事

**だから：**
- 見積が何回も出る
- 追加が出る
- 判断が揺れる
- フロントが疲弊する

### システム再整理案

#### ① 車検には「3つの箱」だけあればいい

**箱①: 法定必須（Must）**
- 車検NG、やらない選択肢なし

**箱②: 車検推奨（Should）**
- 今やるべき、車検と一緒が合理的

**箱③: 車検外提案（Optional / Separate）**
- 別案件、別予算、別判断
- **今の「追加見積内容」は③に特化させるべき**

#### ② 「見積回数が増える」のは異常ではない

**見積 v1（事前）**  
**見積 v2（受入れ）**  
**見積 v3（作業中）**

これは**履歴として残すべき価値ある情報**。

#### ③ 「新しい作業」とは何か？（定義）

**新しい作業とは：**
- 判断軸が変わる
- 説明が変わる
- 承諾の意味が変わる

**例：**
- 車検 → 安全・法令
- エアコン → 快適・予算
- **同じ見積に入れると、話が壊れる**

### 最終的な提言

**車検DXの目的は：**

> 「作業を管理する」ではなく  
> 「判断の文脈を分けて、混乱を減らす」

**そのために必要なもの：**
- 車検という"器"は1つ
- その中に：法定、推奨
- そこから外に：別作業（新しい仕事）

## ✅ レビュー結果

### 1. 見解の評価

#### 優れている点

1. **業務フローの本質を捉えている**
   - 「3つの入口 × 2つの判断 × 2つの分岐」というフレームワークは非常に明確
   - 整備工場の業務を「車を直す」ではなく「意思決定を支援する」と定義している点が本質的

2. **顧客パターンの分類が実用的**
   - パターンC（困りごと起点）は私の提案書では見落としていた重要なパターン
   - 3つのパターンでほぼ全てのケースをカバーできる

3. **「新しい作業」の定義が明確**
   - 「判断軸が変わる」「説明が変わる」「承諾の意味が変わる」という定義は非常に実用的
   - 同じ見積に入れると話が壊れるという指摘は重要

4. **見積回数が増えることを正常な業務フローとして認識**
   - 私の提案書では「問題」として扱っていたが、これは正常な業務フロー
   - 履歴として残すべき価値ある情報という認識は正しい

5. **「追加見積内容」の役割を明確化**
   - 「車検外提案（Optional / Separate）」に特化させるべきという提案は実用的
   - 3つの箱（法定必須、車検推奨、車検外提案）という整理は明確

#### 改善が必要な点

1. **実装の詳細が不足**
   - フレームワークは優れているが、具体的なUI/UXの実装方法が不明確
   - システムの既存実装との整合性が確認できていない

2. **見積履歴の管理方法が不明確**
   - 「見積 v1、v2、v3」を履歴として残すべきという提案は正しいが、具体的な実装方法が不明
   - 現在のシステムに見積履歴機能があるかどうか確認が必要

3. **「新しい作業」の追加方法が不明確**
   - 「新しい作業を追加」ボタンの具体的な実装方法が不明
   - 現在のシステムで「新しい作業」を追加する機能があるかどうか確認が必要

## 🔄 提案書の改善点

### 1. 業務フローの理解を深める

**追加すべき内容:**
- 3つの入口パターン（即入庫、事前見積、困りごと起点）の説明
- 2種類の判断（法的に必須 vs 商売として提案）の説明
- 2つの分岐（車検枠内 vs 車検外）の説明

### 2. 「追加見積内容」の役割を再定義

**変更前:**
- 「車検の点検項目に該当しない、新しく発見された問題や追加作業を記録する」

**変更後:**
- **「車検外提案（Optional / Separate）」に特化**
- 別案件、別予算、別判断として扱うべき問題を記録する
- 判断軸・説明・承諾の意味が変わる問題を記録する

### 3. 「新しい作業」の定義を明確化

**追加すべき内容:**
- 「新しい作業」とは、判断軸が変わる、説明が変わる、承諾の意味が変わるもの
- 例：車検（安全・法令）vs エアコン（快適・予算）
- 同じ見積に入れると話が壊れるという理由

### 4. 見積回数が増えることを正常な業務フローとして認識

**追加すべき内容:**
- 見積回数が増えるのは異常ではない
- 見積 v1（事前）、v2（受入れ）、v3（作業中）は履歴として残すべき価値ある情報
- 見積履歴機能の実装を検討

### 5. 3つの箱の整理

**追加すべき内容:**
- 箱①: 法定必須（Must）- 車検NG、やらない選択肢なし
- 箱②: 車検推奨（Should）- 今やるべき、車検と一緒が合理的
- 箱③: 車検外提案（Optional / Separate）- 別案件、別予算、別判断
