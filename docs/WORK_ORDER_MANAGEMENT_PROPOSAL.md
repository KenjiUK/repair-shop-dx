## 1. 問題の整理

### 1-1. 現状の課題

1. **途中で作業が追加される**: 車検で入庫したが、車検後に一緒にオイル交換やコーティングもお願いされたりする
2. **入庫区分≠最終的な作業**: 入庫区分（車検）と最終的な作業（車検+オイル交換+コーティング）が一致しない
3. **請求書の統合**: 入庫単位のJOBをすべて紐づけて出庫時にまとめて請求書をあげる
4. **長期カスタムの管理**: 長期で色々カスタムする場合、入庫単位の車両で管理したい

### 1-2. 他社のベストプラクティス

#### 1-2-1. 国内整備工場システム

**調査結果**:
- **入庫単位での管理**: 1つの入庫に対して複数の作業を紐付ける
- **ワークオーダー（作業指示書）の概念**: 各作業を個別のワークオーダーとして管理し、1つの入庫に紐付ける
- **請求書の統合**: 複数のワークオーダーを1つの請求書にまとめる
- **作業の追加**: 途中で作業を追加しても、既存の入庫に紐付ける

**代表的なシステム**:
- **totoco（トトコ）**: 入庫ごとに複数の作業を管理
- **CarRide（カーライド）**: 複数の請求書をまとめて合計請求書を作成
- **EBE整備システム**: 入庫管理と作業管理を分離、LINE連携、Web予約、作業進捗ボード機能
- **GATCH（ガッチ）**: 入庫予約機能、代車管理機能、作業工程管理

## 2. 提案する設計アーキテクチャ

### 2-1. 3層構造の設計

```
┌─────────────────────────────────────┐
│  入庫（Check-in / Job）              │
│  - 1つの車両が1回入庫する単位        │
│  - 入庫日時、顧客、車両を管理        │
│  - 出庫時にまとめて請求書を発行     │
└─────────────────────────────────────┘
           │
           ├─── ワークオーダー1（車検）
           ├─── ワークオーダー2（オイル交換）← 途中で追加
           └─── ワークオーダー3（コーティング）← 途中で追加
```

### 2-2. データモデル

#### A. 入庫（Job / Check-in）
- **役割**: 1つの車両が1回入庫する単位
- **管理項目**: 
  - 入庫日時
  - 顧客ID
  - 車両ID
  - 出庫日時
  - ステータス（入庫中、出庫済み）
  - **関連するワークオーダーIDのリスト**

#### B. ワークオーダー（Work Order）
- **役割**: 各作業（車検、オイル交換、コーティングなど）を個別に管理
- **管理項目**:
  - ワークオーダーID
  - **入庫ID（Job ID）** - どの入庫に紐づくか
  - 入庫区分（車検、オイル交換、コーティングなど）
  - ステータス（診断中、見積作成待ち、作業待ち、完了など）
  - 診断情報
  - 見積情報
  - 作業情報
  - 費用情報

#### C. 請求書（Invoice）
- **役割**: 1つの入庫に紐づくすべてのワークオーダーをまとめて請求
- **管理項目**:
  - 請求書ID
  - **入庫ID（Job ID）** - どの入庫に紐づくか
  - **ワークオーダーIDのリスト** - 請求対象のワークオーダー
  - 合計金額
  - 請求日時

## 3. 提案する設計案

### 案1: ワークオーダーを独立したエンティティとして管理（推奨）

#### 設計概要

**Zoho CRMの構造**:
- **入庫管理（CustomModule2）**: 入庫単位を管理（既存）
- **ワークオーダー（新規モジュールまたはCustomModule2の子レコード）**: 各作業を個別に管理
- **請求書（新規モジュール）**: 入庫単位で請求書を管理

**データフロー**:
```
1. 入庫: Jobレコードを作成（入庫区分: 車検）
2. ワークオーダー1作成: 車検のワークオーダーを作成し、Jobに紐付け
3. 途中で追加: オイル交換のワークオーダー2を作成し、同じJobに紐付け
4. 途中で追加: コーティングのワークオーダー3を作成し、同じJobに紐付け
5. 出庫時: Jobに紐づくすべてのワークオーダーをまとめて請求書を作成
```

#### メリット
- ✅ **柔軟性**: 途中で作業を追加しても、既存の入庫に紐付け可能
- ✅ **管理性**: 各作業を個別に管理できる
- ✅ **拡張性**: 長期カスタムも同じ構造で管理可能
- ✅ **請求書の統合**: 1つの入庫に紐づくすべてのワークオーダーをまとめて請求可能

#### デメリット
- ⚠️ **複雑性**: データ構造が複雑になる
- ⚠️ **実装コスト**: 新規モジュールまたは子レコードの実装が必要

### 案2: Jobレコード内で複数の作業を管理（シンプル）

#### 設計概要

**Zoho CRMの構造**:
- **入庫管理（CustomModule2）**: 入庫単位を管理（既存）
- **作業リスト（CustomModule2内のMulti-SelectまたはJSON）**: 複数の作業を1つのJobレコード内で管理

**データフロー**:
```
1. 入庫: Jobレコードを作成（入庫区分: 車検）
2. 途中で追加: 同じJobレコードに「オイル交換」を追加
3. 途中で追加: 同じJobレコードに「コーティング」を追加
4. 出庫時: Jobレコード内のすべての作業をまとめて請求書を作成
```

#### メリット
- ✅ **シンプル**: 既存の構造を活用できる
- ✅ **実装コスト**: 新規モジュールが不要

#### デメリット
- ⚠️ **柔軟性**: 各作業を個別に管理しにくい
- ⚠️ **拡張性**: 長期カスタムの管理が難しい

### 案3: ハイブリッド（推奨）

#### 設計概要

**Zoho CRMの構造**:
- **入庫管理（CustomModule2）**: 入庫単位を管理（既存）
  - `field_service_kinds` (Multi-Select): 実施する作業のリスト（車検、オイル交換、コーティングなど）
  - `field_work_orders` (JSONまたは関連レコード): 各作業の詳細情報
- **ワークオーダー詳細**: 各作業の診断・見積・作業情報をJSONまたは関連レコードで管理

**データフロー**:
```
1. 入庫: Jobレコードを作成（入庫区分: 車検）
2. ワークオーダー1作成: 車検の詳細情報をJobレコード内に追加
3. 途中で追加: オイル交換をfield_service_kindsに追加し、詳細情報を追加
4. 途中で追加: コーティングをfield_service_kindsに追加し、詳細情報を追加
5. 出庫時: Jobレコード内のすべての作業をまとめて請求書を作成
```

#### メリット
- ✅ **バランス**: シンプルさと柔軟性のバランスが良い
- ✅ **実装コスト**: 既存の構造を拡張するだけ
- ✅ **管理性**: 各作業を個別に管理できる

#### デメリット
- ⚠️ **データ構造**: JSONまたは関連レコードの管理が必要

### 4-2. 業務フロー

#### シナリオ1: 車検で入庫、途中でオイル交換とコーティングを追加


### 5-1. データモデルの拡張

**ZohoJob型の拡張**:
```typescript
export interface ZohoJob {
  // 既存のフィールド
  id: string;
  field5: JobStage;
  field4: string; // 顧客名Lookup
  field6: string; // 車両ID Lookup
  field22: string; // 入庫日時
  
  // 新規フィールド
  /** 実施作業リスト */
  field_service_kinds?: string[]; // Multi-Select: ["車検", "オイル交換", "コーティング"]
  
  /** ワークオーダー詳細（JSON形式） */

  field_previous_job_id?: string; // Text: 前回のJob ID

/**
 * ワークオーダー
 */
export interface WorkOrder {
  /** ワークオーダーID */
  id: string;
  /** 入庫区分 */
  serviceKind: ServiceKind;
  /** ステータス */
  status: WorkOrderStatus;
  /** 診断情報 */
  diagnosis?: DiagnosisInfo;
  /** 見積情報 */
  estimate?: EstimateInfo;
  /** 作業情報 */

  /** 作成日時 */
  createdAt: Date;
  /** 完了日時 */
  completedAt?: Date;

/**
 * ワークオーダーステータス
 */
export type WorkOrderStatus =

  | "見積作成待ち"
  | "顧客承認待ち"
  | "部品取り寄せ中"
  | "作業待ち"
  | "作業中"
  | "完了";
```

```typescript
  // 3. 各ワークオーダーの費用を合計

}
```

### 5-3. UI実装

**データフロー**:
```
1. 入庫: Jobレコードを作成（入庫区分: 車検）
2. ワークオーダー1作成: 車検のワークオーダーを作成し、Jobに紐付け
3. 診断ビデオ: 診断内容をビデオで記録し、ワークオーダー1に紐付け
4. 途中で追加: オイル交換のワークオーダー2を作成し、同じJobに紐付け
5. 途中で追加: コーティングのワークオーダー3を作成し、同じJobに紐付け
6. 出庫時: Jobに紐づくすべてのワークオーダーをまとめて請求書を作成
```

**メリット**:
- ✅ **次世代システムとの整合性**: Orderry、Shopmanager、Torque360などと同様の設計パターン
- ✅ **完全な独立性**: 各作業を完全に独立して管理可能
- ✅ **拡張性**: ビデオコミュニケーション、顧客ポータルなどの機能追加が容易
- ✅ **スケーラビリティ**: 大量のワークオーダーを効率的に管理可能
- ✅ **監査性**: 各ワークオーダーの変更履歴を個別に追跡可能

**デメリット**:
- ⚠️ **実装コスト**: 新規モジュールの実装が必要
- ⚠️ **APIの複雑化**: 関連レコードの管理が複雑
- ⚠️ **パフォーマンス**: 大量のワークオーダーがある場合のクエリ最適化が必要
