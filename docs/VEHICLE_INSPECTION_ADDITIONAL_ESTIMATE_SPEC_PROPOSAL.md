# 車検の「追加見積もり内容」仕様再検討提案

**作成日**: 2025-01-XX  
**目的**: 車検の受入れ点検における「追加見積もり内容」セクションの役割を明確化

---

## 📋 整備工場の業務フロー理解

### 業務の本質

**整備工場の業務は、実はこうです：**

> 「車を直す」ではなく  
> 「お客さんの意思決定を、段階的に支援する」仕事

**だから：**
- 見積が何回も出る（これは正常な業務フロー）
- 追加が出る
- 判断が揺れる
- フロントが疲弊する

### 整備工場の業務フレームワーク

**整備工場の業務は「3つの入口 × 2つの判断 × 2つの分岐」で説明できます。**

#### ① お客さんの「入庫前パターン」は最低3つある

**パターンA: 「とにかく車検を通してほしい」型（即入庫）**
- 典型的な会話: 「車検お願いします。あとは見てください」
- 業務フロー: 受付 → 法定車検前提の最低限見積（仮） → 入庫・受入れ点検 → 問題が出たら法定NG/任意 → 電話・LINEで承諾 → 作業・完成
- **この人は「決断を工場に委ねている」**

**パターンB: 「事前に見積が欲しい」型（仮チェックあり）**
- 典型的な会話: 「いくらかかるか見てから決めたい」
- 業務フロー: 受付 → 事前点検（簡易 or 仮） → 見積提示 → 承諾 → 入庫・作業 → 作業中に問題発見 → 追加見積 → 承諾
- **「2回目・3回目の見積」が起こる（これは正常な業務フロー）**

**パターンC（重要）: 「困りごと起点」型（車検はきっかけ）**
- 典型的な会話: 「車検もだけど、最近◯◯が気になる」（エアコン、異音、ハンドルブレ、警告灯）
- **車検は"入口"で、本当の仕事は別**

#### ② 整備工場の中では「2種類の判断」が常に走っている

**判断①: これは「法的に必須」か？**
- 車検不適合、保安基準NG、通らない
- **お客さんの同意は形式的（やらない選択肢がない）**

**判断②: これは「商売として提案」か？**
- 今は通る、でも危ない・近い・次はダメ
- **お客さんの意思決定が必要**

#### ③ さらに「2つの分岐」がある（ここが混乱の元）

**分岐A: 車検という枠の中で完結させるか？**
- ブレーキ、タイヤ、オイル、ワイパー
- **車検と一体で説明できる**

**分岐B: 車検とは切り分けるべきか？**
- エアコン、電装、内装、快適装備
- **説明・予算・判断軸が別**

## 📋 現状の実装状況

### 1. 車検の診断フロー

1. **受付（Phase 1）**: 顧客情報、車両情報、走行距離などを入力
2. **受入れ点検（Phase 2 - 診断画面）**: 
   - 点検項目のチェック（6カテゴリ、50項目以上）
   - 各項目のステータス記録（レ=良好、×=交換、△=修理、A=調整など）
   - **追加見積内容セクション**: 手動で「今回絶対必要」「やったほうがいい」「お客さん次第」の3分類で入力
3. **見積作成（Phase 3）**: 
   - 点検項目で問題があった項目（×、△、Aなど）は**自動的に見積項目に追加**される
   - 「追加見積内容」セクションで入力した項目も見積項目に追加される

### 2. 現在の問題点

**ユーザーの疑問:**
- 通常点検項目で問題があった場合、その内容が自動的に見積項目に追加される
- しかし、「追加見積もり内容」セクションも手動で入力できる
- **この2つの役割が重複しているのではないか？**

**業務フロー上の混乱:**
- 点検項目で発見された問題を「追加見積内容」に入力すべきか？
- 車検とは関係ない問題を「追加見積内容」に入力すべきか？
- 新しい作業を追加する場合は「追加見積内容」からできるべきか？
- **見積回数が増えることを「問題」として扱っているが、これは正常な業務フローではないか？**

## 🎯 整備工場の業務フロー分析

### ケース1: 車検の点検項目で発見された問題

**例:**
- ブレーキパッドの摩耗（×=交換）
- タイヤの溝の深さ不足（×=交換）
- エンジンオイルの劣化（×=交換）
- ブレーキフルードの劣化（×=交換）

**現在の処理:**
- 点検項目のステータス（×、△、Aなど）で記録
- 見積画面で自動的に見積項目に追加される
- **「追加見積内容」セクションには入らない**

**問題点:**
- ユーザーが「追加見積内容」に入力してしまう可能性がある
- 重複入力のリスクがある

### ケース2: 車検とは関係ない、新しく発見された問題

**例:**
- エアコンの不具合（冷えない、異音がする）
- ナビゲーションの故障（画面が映らない）
- ドアの開閉不良
- シートの破損

**現在の処理:**
- 「追加見積内容」セクションで手動入力
- または、新しい作業として追加（案件カードの下に各作業単独でワークフローが動く）

**問題点:**
- 「追加見積内容」に入力すべきか、新しい作業として追加すべきか判断が難しい
- 仕様が不明確

### ケース3: 車検整備をしている中で発見された問題

**例:**
- 車検整備中にエンジンの異音を発見
- 車検整備中にバッテリーの劣化を発見
- 車検整備中にタイヤの異常摩耗を発見

**現在の処理:**
- 点検項目に該当する場合は、点検項目のステータスで記録
- 点検項目に該当しない場合は、「追加見積内容」で入力

**問題点:**
- 点検項目に該当するかどうかの判断が難しい
- 車検整備中に発見された問題は、点検項目として記録すべきか、「追加見積内容」として記録すべきか不明確

## 💡 提案: 車検の「3つの箱」の整理

### 車検DXの目的

**車検DXの目的は：**

> 「作業を管理する」ではなく  
> 「判断の文脈を分けて、混乱を減らす」

**そのために必要なもの：**
- 車検という"器"は1つ
- その中に：法定、推奨
- そこから外に：別作業（新しい仕事）

### 車検には「3つの箱」だけあればいい

**箱①: 法定必須（Must）**
- 車検NG、やらない選択肢なし
- **判断①: 法的に必須**
- **分岐A: 車検という枠の中で完結**

**箱②: 車検推奨（Should）**
- 今やるべき、車検と一緒が合理的
- **判断②: 商売として提案**
- **分岐A: 車検という枠の中で完結**

**箱③: 車検外提案（Optional / Separate）**
- 別案件、別予算、別判断
- **判断②: 商売として提案**
- **分岐B: 車検とは切り分けるべき**
- **今の「追加見積内容」は③に特化させるべき**

## 💡 提案: 仕様の明確化

### 「追加見積内容」の役割を再定義

**「追加見積内容」セクションの役割:**
- **「車検外提案（Optional / Separate）」に特化**
- 別案件、別予算、別判断として扱うべき問題を記録する
- **判断軸・説明・承諾の意味が変わる問題を記録する**

**具体的な使用例:**
1. **エアコンの不具合**: 判断軸が「快適・予算」に変わる → 「追加見積内容」で入力
2. **ナビゲーションの故障**: 判断軸が「快適・予算」に変わる → 「追加見積内容」で入力
3. **ドアの開閉不良**: 判断軸が「快適・予算」に変わる → 「追加見積内容」で入力
4. **シートの破損**: 判断軸が「快適・予算」に変わる → 「追加見積内容」で入力

**点検項目で発見された問題:**
- **点検項目のステータス（×、△、Aなど）で記録する**
- **「追加見積内容」には入力しない**
- 見積画面で自動的に見積項目に追加される
- **判断軸は「安全・法令」のまま（車検と一体で説明できる）**

### 「新しい作業」の定義を明確化

**「新しい作業」とは：**
- **判断軸が変わる**
- **説明が変わる**
- **承諾の意味が変わる**

**例：**
- 車検 → 安全・法令
- エアコン → 快適・予算
- **同じ見積に入れると、話が壊れる**

**「新しい作業を追加」ボタンの役割:**
- **車検とは関係ない、新しい作業を追加するためのエントリーポイント**
- ボタンをクリックすると、新しい作業（ワークオーダー）が作成される
- 新しい作業は、案件カードの下に各作業単独でワークフローが動く

**具体的な使用例:**
1. **エアコンの不具合**: 「新しい作業を追加」ボタンをクリック → 「エアコン修理」という新しい作業が作成される（判断軸: 快適・予算）
2. **ナビゲーションの故障**: 「新しい作業を追加」ボタンをクリック → 「ナビゲーション修理」という新しい作業が作成される（判断軸: 快適・予算）

### 見積履歴の管理

**「見積回数が増える」のは異常ではない**

**見積 v1（事前）**  
**見積 v2（受入れ）**  
**見積 v3（作業中）**

これは**履歴として残すべき価値ある情報**。

**UI改善提案:**
- 見積履歴機能を実装（見積 v1、v2、v3を履歴として表示）
- 各見積の変更内容を記録
- 見積の変更理由を記録

**UI改善提案:**
- 「追加見積内容」セクションのタイトルを「車検外提案（別案件・別予算・別判断）」に変更
- 「新しい作業を追加」ボタンを追加
- 説明文を追加：
  - 「車検とは関係ない、判断軸が変わる問題を記録してください。」
  - 「判断軸が変わる問題（例: エアコン→快適・予算）は『新しい作業を追加』ボタンで追加してください。」
  - 「判断軸が変わらない問題（例: ブレーキ→安全・法令）は、点検項目のステータスで記録してください。」

## 🎯 推奨仕様

### 1. 車検の「3つの箱」の整理

**箱①: 法定必須（Must）**
- 車検NG、やらない選択肢なし
- **判断①: 法的に必須**
- **分岐A: 車検という枠の中で完結**
- **点検項目のステータス（×、△、Aなど）で記録**
- 見積画面で自動的に見積項目に追加される

**箱②: 車検推奨（Should）**
- 今やるべき、車検と一緒が合理的
- **判断②: 商売として提案**
- **分岐A: 車検という枠の中で完結**
- **点検項目のステータス（×、△、Aなど）で記録**
- 見積画面で自動的に見積項目に追加される

**箱③: 車検外提案（Optional / Separate）**
- 別案件、別予算、別判断
- **判断②: 商売として提案**
- **分岐B: 車検とは切り分けるべき**
- **「追加見積内容」セクションで記録**
- または「新しい作業を追加」ボタンで新しい作業として追加

### 2. 「追加見積内容」セクションの役割（再定義）

**役割:**
- **「車検外提案（Optional / Separate）」に特化**
- 別案件、別予算、別判断として扱うべき問題を記録する
- **判断軸・説明・承諾の意味が変わる問題を記録する**

**使用例:**
- エアコンの不具合（判断軸: 快適・予算）
- ナビゲーションの故障（判断軸: 快適・予算）
- ドアの開閉不良（判断軸: 快適・予算）
- シートの破損（判断軸: 快適・予算）

**使用しない例:**
- ブレーキパッドの摩耗（判断軸: 安全・法令 → 点検項目のステータスで記録）
- タイヤの溝の深さ不足（判断軸: 安全・法令 → 点検項目のステータスで記録）
- エンジンオイルの劣化（判断軸: 安全・法令 → 点検項目のステータスで記録）

### 3. 「新しい作業を追加」ボタンの役割

**役割:**
- **判断軸が変わる、説明が変わる、承諾の意味が変わる問題を新しい作業として追加**
- ボタンをクリックすると、新しい作業（ワークオーダー）が作成される
- 新しい作業は、案件カードの下に各作業単独でワークフローが動く
- **同じ見積に入れると話が壊れる問題を分離**

**使用例:**
- エアコンの不具合（判断軸: 快適・予算 → 新しい作業として追加）
- ナビゲーションの故障（判断軸: 快適・予算 → 新しい作業として追加）
- ドアの開閉不良（判断軸: 快適・予算 → 新しい作業として追加）

### 4. 点検項目のステータスの役割

**役割:**
- **車検の点検項目で発見された問題を記録する**
- **判断軸: 安全・法令（車検と一体で説明できる）**
- 見積画面で自動的に見積項目に追加される
- **「追加見積内容」には入力しない**

**使用例:**
- ブレーキパッドの摩耗（×=交換、判断軸: 安全・法令）
- タイヤの溝の深さ不足（×=交換、判断軸: 安全・法令）
- エンジンオイルの劣化（×=交換、判断軸: 安全・法令）
- ブレーキフルードの劣化（×=交換、判断軸: 安全・法令）

### 5. 見積履歴の管理

**役割:**
- **見積回数が増えることは正常な業務フロー**
- 見積 v1（事前）、v2（受入れ）、v3（作業中）を履歴として記録
- 各見積の変更内容と変更理由を記録
- **履歴として残すべき価値ある情報**
